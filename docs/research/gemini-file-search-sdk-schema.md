# Gemini File Search SDK Schema (`@google/genai`) — Query/Citation Response & Store/Document Metadata

> Deep-dive technical research supporting the Gemini File Search Store Navigator backend
> return types (`QueryResult`/`QuerySource`, `StoreInfo`, `DocumentInfo`), the CLI query
> renderer (answer + sources + citations + excerpts), and the hybrid registry metadata cache.
>
> **SDK researched:** `@google/genai` (npm), repo `googleapis/js-genai`, `main` branch.
> Context7 reports latest indexed version **v2.0.1**. All interface bodies below are quoted
> **verbatim** from the SDK's generated `src/types.ts` (the authoritative TypeScript
> definitions), not from blog posts.
>
> **Date of research:** 2026-06-20.

---

## Overview

The Gemini File Search feature is reached through three method groups on the `@google/genai`
client (`GoogleGenAI` instance, conventionally `ai`):

- **Stores** — `ai.fileSearchStores.{create, list, get, delete}`
- **Documents** — `ai.fileSearchStores.documents.{list, get, delete}` plus
  `ai.fileSearchStores.uploadToFileSearchStore` (and `importFile`) for ingestion
- **Query** — `ai.models.generateContent({ ..., config: { tools: [{ fileSearch: {...} }] } })`,
  with citations surfaced under `response.candidates[0].groundingMetadata`
- **Polling** — `ai.operations.get({ operation })` for the long-running upload/import operation

The single most important finding for this project: **the JS SDK exposes camelCase field
names** (`groundingChunks`, `retrievedContext`, `pageNumber`, `mediaId`, `customMetadata`,
`displayName`, `createTime`, `sizeBytes`, …), in contrast to the REST/Python surface which
uses snake_case (`grounding_chunks`, `retrieved_context`, `page_number`, …). This is settled
authoritatively below (see "camelCase vs snake_case").

A second high-value finding: the **`FileSearchStore` object returns document counts and total
size directly at the store level** (`activeDocumentsCount`, `pendingDocumentsCount`,
`failedDocumentsCount`, `sizeBytes`) — so `getStore` does **not** strictly need a
documents-list pass to populate those numbers (the investigation flagged this as uncertain).
See "Store-level vs computed-from-documents".

## camelCase vs snake_case — settled

| Surface | Field style | Example |
|---|---|---|
| REST API / Python SDK | `snake_case` | `grounding_metadata.grounding_chunks[].retrieved_context.page_number` |
| **`@google/genai` JS SDK** | **`camelCase`** | `groundingMetadata.groundingChunks[].retrievedContext.pageNumber` |

**Authoritative source that settles it:** the SDK's generated `src/types.ts`
(`googleapis/js-genai`, `main`). Every interface field is declared in camelCase. Quoted
directly below — e.g. `GroundingChunkRetrievedContext` declares `pageNumber?: number;`,
`mediaId?: string;`, `customMetadata?: GroundingChunkCustomMetadata[];`, `fileSearchStore?: string;`.
This is generated code (`// Code generated by the Google Gen AI SDK generator DO NOT EDIT.`),
so it is the canonical contract the runtime returns, taking precedence over any blog post.

**Confidence: HIGH.** Confirmed by reading the SDK type source directly, cross-checked against
the official File Search docs (which present the snake_case REST shape) and the AI-SDK Google
provider notes.

> Practical implication: do **not** reach for snake_case keys when reading a JS-SDK response
> object. If you ever proxy raw REST yourself, you must translate. The backend should depend on
> the camelCase shape exclusively.

## Part 1 — Query / Citation Response Shape (highest priority)

### Call shape

```typescript
const response = await ai.models.generateContent({
  model: 'gemini-2.5-flash',
  contents: prompt,
  config: {
    tools: [{
      fileSearch: {
        fileSearchStoreNames: [storeName /*, store2, ... up to 5 */],
        metadataFilter: 'department="Engineering" AND year>=2024', // optional
      },
    }],
  },
});
```

`response` is a `GenerateContentResponse`. The answer text is available via the `response.text`
getter; citations live under `response.candidates[0].groundingMetadata`.

### Verbatim SDK types (the citation chain)

**`Candidate`** (the relevant fields):

```typescript
export declare interface Candidate {
  content?: Content;
  citationMetadata?: CitationMetadata;
  finishReason?: FinishReason;
  /** Output only. Metadata returned when grounding is enabled. It
      contains the sources used to ground the generated content. */
  groundingMetadata?: GroundingMetadata;
  index?: number;
  // ... avgLogprobs, logprobsResult, safetyRatings, urlContextMetadata, finishMessage, tokenCount
}
```

**`GroundingMetadata`**:

```typescript
export declare interface GroundingMetadata {
  imageSearchQueries?: string[];
  /** A list of supporting references retrieved from the grounding source. This field is
      populated when the grounding source is Google Search, Vertex AI Search, or Google Maps. */
  groundingChunks?: GroundingChunk[];
  /** List of grounding support. */
  groundingSupports?: GroundingSupport[];
  retrievalMetadata?: RetrievalMetadata;
  searchEntryPoint?: SearchEntryPoint;

  googleMapsWidgetContextToken?: string;
  retrievalQueries?: string[];          // not supported in Gemini API
  sourceFlaggingUris?: GroundingMetadataSourceFlaggingUri[];

```

For File Search, the populated fields are **`groundingChunks`** and **`groundingSupports`**.
(`webSearchQueries`, `searchEntryPoint`, `imageSearchQueries`, Maps fields are for Google
Search / Maps grounding and stay empty for File Search.)

**`GroundingChunk`** — a union-ish container; for File Search only `retrievedContext` is set:

```typescript
export declare interface GroundingChunk {
  image?: GroundingChunkImage;               // image search
  maps?: GroundingChunkMaps;                 // Google Maps
  /** A grounding chunk from a data source retrieved by a retrieval tool. */
  retrievedContext?: GroundingChunkRetrievedContext;   // <-- File Search
  web?: GroundingChunkWeb;                   // Google Search
}
```

**`GroundingChunkRetrievedContext`** — THE source object for File Search citations:

```typescript
export declare interface GroundingChunkRetrievedContext {
  /** ...full resource name of the referenced Vertex AI Search document...
      This field is not supported in Gemini API. */
  documentName?: string;
  /** ...RAG retrieval result... This field is not supported in Gemini API. */
  ragChunk?: RagChunk;
  /** The content of the retrieved data source. */

  title?: string;

      This field is not supported in Vertex AI. */
  customMetadata?: GroundingChunkCustomMetadata[];
  /** Optional. Name of the `FileSearchStore` containing the document.
      Example: `fileSearchStores/123`. This field is not supported in Vertex AI. */
  fileSearchStore?: string;
  /** Optional. Page number of the retrieved context, if applicable.
      This field is not supported in Vertex AI. */
  pageNumber?: number;
  /** Optional. The media blob resource name for multimodal file search results.
      Format: fileSearchStores/{file_search_store_id}/media/{blob_id}.
      This field is not supported in Vertex AI. */

```

Key takeaways:
- `text` = the excerpt the renderer prints; `title` = the document display name / source label.
- `pageNumber` is a **`number`** (camelCase) when present.
- `mediaId` is the multimodal blob reference; `fileSearchStore` is the **per-chunk store name**
  — this is the field that enables **multi-store attribution** (see below).
- `customMetadata` here is `GroundingChunkCustomMetadata` (key + value), distinct from the
  document-side `CustomMetadata` type (which adds `stringListValue`); see both below.

**`GroundingChunkCustomMetadata`** (response-side):

```typescript
export declare interface GroundingChunkCustomMetadata {
  /** The key of the metadata. */
  key?: string;
  numericValue?: number;            // line elided in quote; present in SDK
  stringListValue?: GroundingChunkStringList;
  /** Optional. The string value of the metadata. */
  stringValue?: string;
}
```

**`GroundingSupport`** — maps a generated answer segment to the chunk indices (inline citations):

```typescript
export declare interface GroundingSupport {
  /** Confidence score of the support references. Ranges 0..1.
      Same size as groundingChunkIndices. */
  confidenceScores?: number[];
  /** A list of indices (into 'groundingChunks') specifying the citations associated with
      the claim. e.g. [1,3,4] => groundingChunks[1], [3], [4] support this claim. */
  groundingChunkIndices?: number[];
  /** Segment of the content this support belongs to. */
  segment?: Segment;
  renderedParts?: number[];
}
```

**`Segment`** — the slice of `response.text` a support refers to:

```typescript
export declare interface Segment {
  /** Output only. Start index in the given Part, measured in BYTES. inclusive, from 0. */
  startIndex?: number;
  /** Output only. End index in the given Part, measured in BYTES. exclusive, from 0. */
  endIndex?: number;
  /** Output only. The index of a Part object within its parent Content object. */
  partIndex?: number;
  /** Output only. The text corresponding to the segment from the response. */
  text?: string;
}
```

> **Critical pitfall — `startIndex`/`endIndex` are BYTE offsets, not character offsets.**
> For inline-citation rendering you must slice on a UTF-8 byte buffer, not on a JS string by
> `.slice()`, or multi-byte characters (emoji, accented letters, CJK) will misalign the
> markers. The safe approach: convert `response.text` to a `Buffer`/`Uint8Array`, splice
> citation markers at the byte offsets, then decode back to a string. The `segment.text` field
> usually carries the exact segment text, so you can also locate/anchor using that as a
> cross-check.

### How `response.text` relates to the supports

- `response.text` is the full generated answer (concatenation of the text parts of
  `candidates[0].content.parts`).
- Each `groundingSupports[i].segment` identifies a byte-range slice of that text (the claim).
- `groundingSupports[i].groundingChunkIndices` lists which `groundingChunks[]` back that claim.
- So inline citations are rendered by walking `groundingSupports`, and for each, appending
  markers like `[1][3]` (1-based or 0-based as you choose) at `segment.endIndex`, where the
  numbers map to `groundingChunkIndices` (deduplicated into a stable source list).

### Behavior when no grounding is returned

`groundingMetadata`, `groundingChunks`, and `groundingSupports` are **all optional**. The model
may answer without retrieving (or retrieve nothing relevant). Robust handling:

- `response.candidates?.[0]?.groundingMetadata` may be `undefined` → treat as "no sources".
- `groundingChunks` may be `undefined` or `[]` → `sources: []`, render answer only.
- `groundingSupports` may be `undefined` even when chunks exist → fall back to a flat source
  list (no inline markers), still printing excerpts.
- `response.text` itself can be empty if `finishReason` is e.g. `SAFETY`/`MAX_TOKENS`; check
  `candidates[0].finishReason` before assuming an answer exists.

Never index `candidates[0]` without optional chaining — `candidates` can be empty when the
prompt is blocked (`response.promptFeedback.blockReason`).

### Multi-store query attribution

- A query may target up to **5 stores** (`fileSearchStoreNames: [...]`).
- Each returned chunk carries its own **`retrievedContext.fileSearchStore`** (e.g.
  `fileSearchStores/abc123`). Use this to attribute each excerpt/source back to its origin
  store when querying multiple stores at once.
- `title` is the document's display name; `fileSearchStore` is the owning store id. Together
  they give "which document in which store".

### Raw `--json` passthrough shape

For a faithful `--json` dump, serialize **`response.candidates[0].groundingMetadata`** (the
whole object) plus `response.text`. That object is plain JSON-serializable (no class instances
in the grounding subtree). A good envelope:

```jsonc
{
  "answer": "<response.text>",
  "groundingMetadata": { /* candidates[0].groundingMetadata verbatim */ },
  "finishReason": "STOP",
  "usageMetadata": { /* response.usageMetadata if useful */ }
}
```

> Note: top-level `GenerateContentResponse` carries some class instances (e.g.
> `usageMetadata`, `promptFeedback`) and an `sdkHttpResponse`. Prefer serializing the specific
> sub-objects you need rather than `JSON.stringify(response)` wholesale, which can include the
> raw HTTP response. The `groundingMetadata` subtree itself is all plain interfaces.

### Worked example — extract answer + sources + excerpts + inline citations

```typescript
import { GoogleGenAI } from '@google/genai';

const ai = new GoogleGenAI({ apiKey });  // explicit key — see "API-key precedence" pitfall

const response = await ai.models.generateContent({
  model: 'gemini-2.5-flash',
  contents: prompt,
  config: {
    tools: [{ fileSearch: { fileSearchStoreNames: storeNames } }],
  },
});

const answer = response.text ?? '';
const candidate = response.candidates?.[0];
const gm = candidate?.groundingMetadata;
const chunks = gm?.groundingChunks ?? [];
const supports = gm?.groundingSupports ?? [];

// 1) Build the deduplicated source/excerpt list from retrievedContext.
interface QuerySource {
  title?: string;
  excerpt?: string;
  storeName?: string;     // retrievedContext.fileSearchStore — multi-store attribution
  pageNumber?: number;
  mediaId?: string;
  uri?: string;
  customMetadata?: Array<{ key?: string; stringValue?: string; numericValue?: number }>;
}

const sources: QuerySource[] = chunks.map((c) => {
  const rc = c.retrievedContext;
  return {
    title: rc?.title,
    excerpt: rc?.text,
    storeName: rc?.fileSearchStore,
    pageNumber: rc?.pageNumber,
    mediaId: rc?.mediaId,
    uri: rc?.uri,
    customMetadata: rc?.customMetadata?.map((m) => ({
      key: m.key, stringValue: m.stringValue, numericValue: m.numericValue,
    })),
  };
});

// 2) Inline citations: splice [n] markers into the answer at segment byte offsets.
//    BYTE-accurate slicing (segment.startIndex/endIndex are byte offsets).
function renderInlineCitations(text: string, supports: GroundingSupportLike[]): string {
  const bytes = Buffer.from(text, 'utf-8');
  // Process supports in descending endIndex order so earlier offsets stay valid.
  const ordered = [...supports]
    .filter((s) => s.segment?.endIndex != null && s.groundingChunkIndices?.length)
    .sort((a, b) => (b.segment!.endIndex! - a.segment!.endIndex!));

  let out = bytes;
  for (const s of ordered) {
    const marker = Buffer.from(
      ' ' + s.groundingChunkIndices!.map((i) => `[${i + 1}]`).join(''),  // 1-based labels
      'utf-8',
    );
    const at = s.segment!.endIndex!;
    out = Buffer.concat([out.subarray(0, at), marker, out.subarray(at)]);
  }
  return out.toString('utf-8');
}

const answerWithCitations = supports.length
  ? renderInlineCitations(answer, supports)
  : answer;   // graceful fallback when no supports

// Minimal structural type to avoid importing SDK internals in the helper:
type GroundingSupportLike = {
  segment?: { startIndex?: number; endIndex?: number; text?: string };
  groundingChunkIndices?: number[];
  confidenceScores?: number[];
};
```

This yields the three render sections the skill/CLI produce — ANSWER (`answerWithCitations`),
SOURCES (`sources[].title` + `storeName`), EXCERPTS (`sources[].excerpt`) — plus inline `[n]`
markers tied to `groundingChunks` indices.

## Part 2 — Store Metadata (`fileSearchStores.get` / `list`)

**`FileSearchStore`** (verbatim):

```typescript
export declare interface FileSearchStore {
  /** Output only. Immutable. Identifier. The `FileSearchStore` resource name...
      Example: `fileSearchStores/my-awesome-file-search-store-123a456b789c` */
  name?: string;
  /** Optional. Human-readable display name (<= 512 chars). */
  displayName?: string;
  /** Output only. Timestamp of creation. */
  createTime?: string;
  /** Output only. Timestamp of last update. */
  updateTime?: string;
  /** Output only. Number of documents that are active and ready for retrieval. */
  activeDocumentsCount?: string;
  /** Output only. Number of documents being processed. */
  pendingDocumentsCount?: string;
  /** Output only. Number of documents that have failed processing. */
  failedDocumentsCount?: string;
  /** Output only. Total size of raw bytes ingested across all documents. */
  sizeBytes?: string;
  /** The embedding model used by the FileSearchStore. */
  embeddingModel?: string;
}
```

Notes:
- **`name`** is the API identifier (`fileSearchStores/...`); there is **no separate `state`
  field** on a store. "State" is expressed via the three count fields.
- **Counts and `sizeBytes` are `string`** (int64-as-string JSON convention). Parse with
  `Number(...)`/`BigInt(...)` before arithmetic.
- `createTime`/`updateTime` are RFC 3339 timestamp strings.
- `embeddingModel` (format `models/{model}`) is settable at create; otherwise default.

**List shape** — `ai.fileSearchStores.list({ config: { pageSize } })` returns a **pager**; the
underlying response class is:

```typescript
export class ListFileSearchStoresResponse {
  sdkHttpResponse?: HttpResponse;
  nextPageToken?: string;
  fileSearchStores?: FileSearchStore[];
}
```

Config: `ListFileSearchStoresConfig { pageSize?: number; pageToken?: string; httpOptions?; abortSignal? }`.

**Get shape** — `ai.fileSearchStores.get({ name })` returns a `FileSearchStore`.
Config: `GetFileSearchStoreConfig { httpOptions?; abortSignal? }`.

**Create** — `ai.fileSearchStores.create({ config: { displayName, embeddingModel? } })` returns
a `FileSearchStore`.

**Delete** — `ai.fileSearchStores.delete({ name, config: { force? } })`; `force: true` also
deletes contained documents.

**`DocumentState`** enum (string values equal to member names):

```typescript
enum DocumentState {
  STATE_UNSPECIFIED = 'STATE_UNSPECIFIED',  // default/omitted
  STATE_PENDING     = 'STATE_PENDING',      // chunks being embedded/stored
  STATE_ACTIVE      = 'STATE_ACTIVE',       // processed, queryable
  STATE_FAILED      = 'STATE_FAILED',       // some chunks failed
```

> The skill's renderer prints `ACTIVE` / `PENDING` / `FAILED`; the SDK enum is
> `STATE_ACTIVE` / `STATE_PENDING` / `STATE_FAILED`. When porting, either keep the SDK form or
> strip the `STATE_` prefix for display — but the backend `DocumentInfo.state` should store the
> raw SDK value to avoid lossy mapping.

```typescript
  stringValue?: string;
```

`sizeBytes` here is again a **`string`** (int64-as-string).

**Delete** — `ai.fileSearchStores.documents.delete({ name, config: { force? } })`.

### Upload / import operation shape and async polling contract

`ai.fileSearchStores.uploadToFileSearchStore({ file, fileSearchStoreName, config })` and
`ai.fileSearchStores.importFile({ fileSearchStoreName, fileName, config })` are **long-running
operations**. They return an `ImportFileOperation`:

```typescript
export class ImportFileOperation implements Operation<ImportFileResponse> {
  name?: string;
  metadata?: Record<string, unknown>;
  /** false = still in progress; true = done (then error or response is set). */
  done?: boolean;
  error?: Record<string, unknown>;
  response?: ImportFileResponse;   // available when done
  sdkHttpResponse?: HttpResponse;
}

export class ImportFileResponse {
  sdkHttpResponse?: HttpResponse;
  /** The name of the FileSearchStore containing Documents. */
  parent?: string;
  /** The identifier for the Document imported. */
  documentName?: string;
}
```

**Polling contract** (matches the skill's proven loop):

```typescript
let operation = await ai.fileSearchStores.uploadToFileSearchStore({
  file: filePath,
  fileSearchStoreName: storeName,
  config: { displayName, mimeType, customMetadata, chunkingConfig },
});

while (!operation.done) {
  await new Promise((r) => setTimeout(r, 2000));      // 1–2s backoff is typical
  operation = await ai.operations.get({ operation }); // pass the operation object
}

if (operation.error) {
  throw new Error(`Upload failed: ${JSON.stringify(operation.error)}`);
}
const importedDocName = operation.response?.documentName;   // resource name of the new Document
```

> **Important:** the operation result is an **`ImportFileResponse`** carrying only `parent` and
> `documentName` — **not** a full `Document`. To return a populated `DocumentInfo` from
> `uploadDocument`, follow the poll with a
> `ai.fileSearchStores.documents.get({ name: operation.response.documentName })` to fetch
> `state`, `sizeBytes`, `mimeType`, timestamps, etc. (A freshly uploaded document is commonly
> `STATE_PENDING` immediately after the operation reports `done`; embedding continues
> server-side. Decide whether `uploadDocument` waits for `STATE_ACTIVE` or returns at
> `STATE_PENDING`.)

**`UploadToFileSearchStoreConfig`** fields: `mimeType?`, `displayName?`,
`customMetadata?: CustomMetadata[]`, `chunkingConfig?: ChunkingConfig`, plus `httpOptions?`/`abortSignal?`.
`ChunkingConfig { whiteSpaceConfig?: { maxTokensPerChunk?: number; maxOverlapTokens?: number } }`.

### Pagination contract (`pager.page`, `hasNextPage()`, `nextPage()`)

Both `fileSearchStores.list` and `documents.list` return an async **Pager** object (not the
raw response). Observed usage (from the working skill and SDK conventions):

```typescript
const pager = await ai.fileSearchStores.list({ config: { pageSize: 20 } });

// Option A: iterate the current page array directly
for (const store of pager.page) { /* ... */ }

// Option B: walk all pages manually
let page = pager.page;
while (true) {
  for (const store of page) { /* ... */ }
  if (!pager.hasNextPage()) break;
  page = await pager.nextPage();
}

// Option C: async-iterate the whole collection (documents)
const docPager = await ai.fileSearchStores.documents.list({ parent: storeName });
for await (const doc of docPager) { /* doc: Document */ }
```

- `pager.page` → current page array (`FileSearchStore[]` or `Document[]`).
- `pager.hasNextPage()` → boolean.
- `pager.nextPage()` → Promise resolving to the next page array (advances internal cursor).
- The raw `nextPageToken` is available on the underlying `List*Response` if you need
  token-based pagination for the backend's `Page<T>.nextPageToken`. To surface a page token in
  an `IGeminiBackend.listStores` that takes `{ pageSize, pageToken }`, pass `pageToken` through
  `config` and read `nextPageToken` from the response; the pager abstracts this for in-process
  iteration but the token is the durable cursor to persist across HTTP requests.

> **Backend note:** the pager is convenient in-process, but the HTTP API surface needs an
> explicit `pageToken` in/`nextPageToken` out. Prefer driving the SDK with explicit
> `config.pageToken` and returning `response.nextPageToken` for the API; use the pager's
> `for await` form for CLI "list everything" convenience.

## Part 4 — Store-level vs computed-from-documents

The investigation flagged uncertainty over whether `documentCount`/`totalSize`/`state` are
store-level or must be computed by listing documents. **Resolved:**

| Field | Source | How to obtain |
|---|---|---|
| `documentCount` (total) | **Store-level (derivable)** | `active + pending + failed` from `FileSearchStore` counts; or a `documents.list` count if you need an exact live tally |
| `activeDocumentsCount` / `pendingDocumentsCount` / `failedDocumentsCount` | **Store-level** | Directly on `FileSearchStore` (as `string`) |
| `totalSize` / `sizeBytes` | **Store-level** | `FileSearchStore.sizeBytes` (as `string`) — total raw bytes ingested |
| store "state" | **Not a field** | Derive from counts (e.g. "all active" / "N pending" / "M failed") |
| per-document `state` | Document-level | `Document.state` (`DocumentState`) |
| per-document `sizeBytes`, `mimeType` | Document-level | `Document` fields |

**Implication for the backend:** `getStore` can populate `documentCount` and `sizeBytes`
**from the store object alone** — no documents-list pass required for the summary numbers. A
documents-list pass is only needed when you want the per-document breakdown (the skill's
`get-store-info` STATISTICS+DOCUMENTS view) or an exact live count that doesn't trust the
(eventually-consistent) store counters. This contradicts the investigation's tentative
assumption (line ~377-379) that count/size "may not be store-level fields"; they **are**
store-level in the current SDK. Treat the counters as possibly slightly stale right after an
upload (server-side aggregation lag) — for an authoritative count immediately post-upload,
list documents.

## Ready-to-drop-in backend TypeScript interfaces

Derived from the confirmed SDK shapes. Designed for `src/core/types.ts` /
`src/core/backend/backend.ts`. Numeric int64-as-string fields are parsed to `number` at the
backend boundary (document them as "may exceed 2^53 only for absurd sizes — acceptable").

```typescript
// ---------- Stores ----------
export interface StoreInfo {
  apiName: string;              // FileSearchStore.name  e.g. "fileSearchStores/abc123"
  displayName?: string;         // FileSearchStore.displayName
  createTime?: string;          // RFC3339
  updateTime?: string;          // RFC3339
  sizeBytes?: number;           // Number(FileSearchStore.sizeBytes)
  activeDocumentsCount?: number;
  pendingDocumentsCount?: number;
  failedDocumentsCount?: number;
  documentCount?: number;       // active + pending + failed (computed)
  embeddingModel?: string;
  // No native "state" — derive a display state from the counts if needed.
}

// ---------- Documents ----------
export type DocumentState =
  | 'STATE_UNSPECIFIED' | 'STATE_PENDING' | 'STATE_ACTIVE' | 'STATE_FAILED' | string;

export interface DocCustomMetadata {
  key: string;
  stringValue?: string;
  numericValue?: number;
  stringListValue?: string[];   // from StringList.values
}

export interface DocumentInfo {
  apiName: string;              // Document.name  e.g. "fileSearchStores/abc/documents/doc1"
  displayName?: string;
  state: DocumentState;         // Document.state (raw SDK value)
  sizeBytes?: number;           // Number(Document.sizeBytes)
  mimeType?: string;
  createTime?: string;
  updateTime?: string;
  customMetadata?: DocCustomMetadata[];
}

// ---------- Query ----------
export interface QuerySource {
  title?: string;               // retrievedContext.title
  excerpt?: string;             // retrievedContext.text
  storeName?: string;           // retrievedContext.fileSearchStore (multi-store attribution)
  uri?: string;                 // retrievedContext.uri
  pageNumber?: number;          // retrievedContext.pageNumber
  mediaId?: string;             // retrievedContext.mediaId
  customMetadata?: Array<{ key?: string; stringValue?: string; numericValue?: number }>;
}

export interface CitationSpan {
  startIndex?: number;          // segment.startIndex (BYTE offset)
  endIndex?: number;            // segment.endIndex (BYTE offset)
  text?: string;                // segment.text
  chunkIndices: number[];       // groundingChunkIndices -> indices into sources[]

}

  answer: string;               // response.text
  sources: QuerySource[];       // from candidates[0].groundingMetadata.groundingChunks

  finishReason?: string;        // candidates[0].finishReason

}
```

## Best Practices

1. **Always optional-chain the citation path.** `response.candidates?.[0]?.groundingMetadata?.groundingChunks ?? []`.
   Every level is optional in the SDK types.
2. **Construct the client with an explicit `{ apiKey }`** from the resolved profile to dodge
   the `GOOGLE_API_KEY` vs `GEMINI_API_KEY` env-precedence quirk (the SDK prefers
   `GOOGLE_API_KEY` when both are set). Explicit key beats the skill's temporary-unset hack.
3. **Parse int64-as-string fields** (`sizeBytes`, `*DocumentsCount`) with `Number()` at the
   backend boundary; never compare them as strings.
4. **Store raw SDK enum values** (`STATE_ACTIVE`, …) in `DocumentInfo.state`; map to display
   form (`ACTIVE`) only in the renderer.
5. **Byte-accurate inline citations.** Slice on a UTF-8 byte buffer for `segment` offsets.
6. **Poll uploads to completion**, then `documents.get(documentName)` to hydrate the full
   `DocumentInfo`; the operation response only returns `parent` + `documentName`.
7. **Persist `nextPageToken`** for the HTTP API; use the pager's `for await` for CLI list-all.
8. **For the metadata cache**, store `StoreInfo` counters + `sizeBytes` + timestamps +
   `lastRefreshedAt`; these come free from `fileSearchStores.get`/`list` without a docs pass.
9. **Multi-store queries:** group/attribute excerpts by `retrievedContext.fileSearchStore`.

## Common Pitfalls

- **Missing `groundingMetadata`** when the model didn't retrieve or was blocked — handle as
  "no sources", and check `finishReason`/`promptFeedback.blockReason`.
- **`segment.startIndex/endIndex` are BYTE offsets**, not char offsets — naive `string.slice`
  corrupts multi-byte text.
- **Upload returns an operation, not a Document** — `done:false` initially; never read
  `response` before `done:true`; check `operation.error`.
- **Freshly uploaded docs are `STATE_PENDING`** even after the op is `done`; decide your
  "wait-for-ACTIVE" policy.
- **`sizeBytes`/counts are strings** → string concatenation bugs if used unparsed.
- **No store-level `state` field** — don't expect one; derive it.
- **`JSON.stringify(response)` leaks `sdkHttpResponse`** — serialize the `groundingMetadata`
  subtree (and chosen fields) for `--json`, not the whole response object.
- **Document immutability** — there is no update; "replace" = delete + re-upload (matches the
  skill). `updateTime` changes on metadata-affecting server events, not content edits.
- **Limits** (from official docs/skill): 100 MB/file, 5 stores/query, 10 stores/project (soft),
  20 custom-metadata entries/document, <20 GB/store recommended — surface as typed errors.

## Assumptions & Scope

- **Interpreted "query/citation response shape" as the File Search grounding path**
  (`retrievedContext`), explicitly distinct from Google Search grounding (`web`) and Maps
  (`maps`) — those share `GroundingMetadata` but populate different chunk subfields. The File
  Search renderer should read only `retrievedContext`.
- **Assumed the `main` branch of `googleapis/js-genai` reflects the version the project will
  pin** (Context7 indexes v2.0.1). The grounding/store/document interfaces are generated and
  stable across recent releases, but pin-time re-verification is advised (Topic 3 / dependency
  vetting).
- **`response.text` getter semantics** assumed to concatenate text parts of `candidates[0]`
  (standard SDK behavior); confirmed by SDK conventions and the skill's working usage.
- **Out of scope:** Vertex AI variant fields (`documentName`, `ragChunk` are "not supported in
  Gemini API"); Google Search / Maps / image grounding subtrees; streaming
  (`generateContentStream`) grounding delivery; the `embeddingModel` selection guidance.

## Assumptions Made

| Assumption | Confidence | Impact if Wrong |
|---|---|---|
| JS SDK uses camelCase across grounding/store/document | HIGH | If wrong, every field read breaks — but confirmed verbatim in generated `types.ts` |
| `FileSearchStore` exposes counts + `sizeBytes` at store level | HIGH | If wrong, `getStore` would need a docs-list pass for stats (confirmed present in SDK) |
| `segment.startIndex/endIndex` are byte offsets | HIGH | If they were char offsets, the byte-buffer slicing would itself misalign; SDK JSDoc explicitly says "measured in bytes" |
| Upload op `response` is `ImportFileResponse` (parent+documentName only) | HIGH | If it returned a full Document, the extra `documents.get` would be redundant (confirmed in SDK) |
| Pager exposes `page` / `hasNextPage()` / `nextPage()` and async iteration | MEDIUM-HIGH | If the pager API differs, list code changes; corroborated by the working skill, not re-read from SDK pager source |
| `documentCount` not a native store field (only the three counts) | HIGH | If a total existed, the `active+pending+failed` sum is still correct |
| Latest pinned version matches `main` interfaces | MEDIUM | A future major could rename fields; re-verify at pin time |

## Uncertainties & Gaps

- **Pager method surface not read from SDK source directly** — `pager.page`/`hasNextPage()`/
  `nextPage()` and `for await` are taken from the working skill (prior art) and SDK
  conventions, not from a quoted `Pager` class. Behavior is reliable in practice; the exact
  method names should be re-confirmed against `src/pagers.ts` when implementing.
- **`response.text` with multiple candidates / multi-part content** — for `candidateCount > 1`
  the renderer must pick `candidates[0]`; multi-part text concatenation order assumed standard.
- **Counter consistency lag** — exactly how stale `activeDocumentsCount` can be immediately
  after an upload is undocumented; for authoritative immediate counts, list documents.
- **`renderedParts` on `GroundingSupport`** and `GroundingMetadata` "rendered_parts" are
  present but undocumented for File Search; not needed for the basic inline-citation renderer.
- **Multimodal `mediaId`** retrieval/rendering (fetching the media blob) was not researched —
  only the field's presence is confirmed.

## Clarifying Questions for Follow-up

1. Should `uploadDocument` block until `STATE_ACTIVE`, or return at `STATE_PENDING` and let the
   caller poll later? (Affects UX latency vs. immediate availability.)
2. For the CLI `--json` query output, is the desired contract the full `groundingMetadata`
   subtree, or the normalized `QueryResult` (sources/citations) shape? (Recommend offering
   both: normalized by default, `--raw-json` for the SDK passthrough.)
3. Should inline-citation markers be 1-based (`[1]`) matching a printed SOURCES list, or 0-based
   matching `groundingChunks` indices? (Example above uses 1-based for human readability.)
4. Does the registry cache need the per-document breakdown, or only store-level
   counts+size+timestamps? (The latter is free from `fileSearchStores.list`; the former costs a
   docs-list per store on refresh.)
